     Equi-recursive computation types for ral

     Context

     The Zydeco paper (zydeco-relmonad.pdf) shows that recursive computation
     types
     ОҪ Y : CTy. B enable powerful patterns in CBPV: variadic functions, stream
     protocols, stack types like Int вҶ’* OS. Currently ral rejects all recursive
     types via occurs checks in unification. The goal is to allow recursive
     computation types while preserving full HM inference and ral's "no type
     annotations" philosophy.

     Equi-recursive types (cycles in the union-find) are the right fit: no new
     syntax (no roll/unroll, no type annotations), decidable with principal
     types
     (Amadio & Cardelli 1993). Recursive value types remain forbidden вҖ” the
     paper's
     ОҪ only binds at kind CTy, and recursive value types (Оұ = [Оұ]) are more
     likely bugs than intent.

     What changes

     All changes are in one file: ral/src/typecheck.rs (plus tests and SPEC).
     No changes to the parser, elaborator, evaluator, or runtime type system.

     Design

     After the change, Comp::Rec at line 1075 does:

     beta = fresh_comp_ty()
     env.bind(name, mono(Thunk(beta)))
     body_ty = infer_comp(body)        -- e.g. Fun(Int, Var(beta))
     unify_comp_ty(body_ty, beta)      -- binds beta = Fun(Int, Var(beta))  вҶҗ
     cycle!

     This produces a cyclic binding in the union-find: comp_ty_slots[N] = 
     Bound(Fun(Int, Var(N))).
     Every function that structurally descends into computation types must
     detect this cycle.

     Value type occurs checks are kept. Value types can still contain recursive
     comp types through Thunk(B), so cycle guards must be threaded through
     the mutual Ty/CompTy traversals.

     Generalization/instantiation correctness

     The hard part. After apply_ty, a cyclic comp type becomes a tree with
     CompTy::Var(root) back-edges. If the cycle body contains a generalized
     ty var Оұ, two instantiations of the scheme would share the same union-find
     slot for the cycle root, but need independent copies with different Оұ
     substitutions.

     Solution: extend Scheme with comp_ty_bindings: Vec<(u32, CompTy)> that
     snapshots the applied binding of each cyclic comp var. instantiate creates
     fresh comp var IDs and re-binds them with the ty/mode/comp var substitution
     applied.

     Implementation steps

     All steps modify ral/src/typecheck.rs unless noted otherwise.

     Step 1: Extend Scheme (mechanical, no behavior change)

     - Add field comp_ty_bindings: Vec<(u32, CompTy)> to Scheme
     - Update Scheme::mono() вҖ” pass comp_ty_bindings: vec![]
     - Update is_poly() вҖ” also check !self.comp_ty_bindings.is_empty()
     - Update seed_builtins (line 1348) вҖ” add comp_ty_bindings: vec![]

     Step 2: Cycle-aware apply_ty / apply_comp_ty (no behavior change yet)

     Add _inner variants that take visited: &mut HashSet<u32> (comp ty var root
     IDs):

     - apply_comp_ty_inner: before recursing into a resolved Var(i), find root.
     If root already in visited вҶ’ return CompTy::Var(root) (cycle back-edge).
     Otherwise insert root, recurse, leave it inserted (don't remove вҖ” the whole
     subtree under that root is already expanded).
     - apply_ty_inner: delegates to apply_comp_ty_inner through Thunk(b).
     - Public apply_ty / apply_comp_ty create empty visited set and delegate.

     Step 3: Cycle-aware ty_occurs_comp (no behavior change yet)

     Step 4: Cycle-aware free-variable collection (no behavior change yet)
