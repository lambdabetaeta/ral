Here's how ral measures against each of the 25 desiderata from the report, followed by suggestions.
1. No implicit word splitting or globbing — HIGH
Verdict: Fully met. Variables never split or glob-expand (§0). $name outside quotes is a separate atom, not a split-and-glob hazard. This is the single most important property and ral nails it, following rc and Fish. Glob is explicit via the glob builtin.
2. Structured data in pipelines — HIGH
Verdict: Substantially met. The pipeline has four stage shapes (Bytes→Bytes, Bytes→Value, Value→Value, Value→Bytes) with explicit adapters (from-X, to-X). This is a cleaner design than Nushell's implicit everything-is-a-table approach because the boundary between bytes and values is always visible. The trade-off is more verbosity — curl -s $url | from-json | map $f rather than just curl $url | map $f — but that's a feature, not a bug, for agent safety. One gap: there's no streaming structured pipeline (value stages are eager), which limits throughput on large datasets.
3. Proper error handling by default — HIGH
Verdict: Fully met. Failure propagation is always on (§10.0). There is no set -e — it's the only mode. try/catch are explicit suppression. guard runs cleanup without suppressing. The ? operator replaces || with clear semantics. The separation of if (Bool) from ? (failure) eliminates the silent-logic-bug class where false is confused with failure. This is one of ral's strongest design points.
4. Static parsing (no re-scanning) — HIGH
Verdict: Fully met. The lexer is a single pass with no modes (§1.1). Input is never rescanned except by eval — and there is no eval. The RATIONALE explicitly cites Shellshock-class vulnerability elimination. ! marks every execution point, making security auditing a grep.
5. Capability-based security / sandboxing — HIGH
Verdict: Deferred, with a clear plan. The RATIONALE lists sandbox as a future item requiring OS-level enforcement (seccomp, App Sandbox). with provides command-level mocking/overriding, which is a form of capability attenuation for testing but not for security enforcement (a malicious script can just not use with). The SHILL-style capability model where scripts receive only the privileges they need is not present in v1. This is the biggest gap relative to the desiderata.
6. Machine-parseable structured output — HIGH
Verdict: Substantially met. _audit returns a full execution tree as JSON (§10.3, §10.8). MCP serialization is specified. ral --audit outputs JSON to stdout. Error records from _try have structured fields (ok, status, cmd, stderr, line, col). The MCP endpoint ral_run enables audit automatically. What's missing: there's no --json flag for individual commands (that's an ecosystem problem, not a shell problem), and exit codes are still integers rather than structured error types — though the _try record compensates.
7. Real type system — HIGH
Verdict: Substantially met. Values include Unit, String, Int, Float, Bool, List, Map, Block, Lambda, Handle (§2). Arithmetic enforces numeric types. Interpolation rejects List/Map/Block/Lambda/Handle. Type mismatches are caught at parse time or runtime with precise error messages. What's missing compared to Nushell: no domain-specific types (filesize, duration, dates). file-size returns an Int in bytes; there's no unit algebra. This is a deliberate simplicity choice, and reasonable for v1.
8. Sane quoting and escaping — HIGH
Verdict: Fully met. Two quoting forms with clean, non-overlapping rules: single quotes (literal, double to embed), double quotes (interpolation with $ and !, seven escape sequences). No backtick escaping. No context-dependent quoting rules. No $IFS. The RATIONALE's path-construction rule ("$dir/file.txt" required, $dir/file is two atoms) makes the safe form the default.
9. Transactional semantics / undo / rollback — HIGH
Verdict: Partially met. guard provides cleanup-on-failure, which is half of transactional semantics (the "finally" half). _audit provides the observability to know what happened. But there is no Start-Transaction / Undo-Transaction equivalent as in PowerShell. No filesystem snapshots, no declarative rollback. For AI agent use, the execution tree gives an agent enough information to decide what to undo, but the shell doesn't do it automatically. This is a meaningful gap for the agentic use case.
10. Sandboxed execution environment — HIGH
Verdict: Not met in v1. Same as #5. The spec acknowledges this explicitly as future work. The with mechanism for command overriding is a building block, and the immutable-bindings / isolated-spawn model reduces the attack surface, but there's no process isolation, network egress control, or filesystem restriction at the shell level.
11. Proper functions with lexical scoping — HIGH
Verdict: Fully met. Lexical scoping with closure capture at definition time (§3). Named parameters via lambda syntax. Currying with partial application (§4.6). Destructuring in parameters (§7). Tail-call optimization guaranteed. The four execution contexts (withdir, withenv, with, audit) are deliberately dynamic but scoped and read-only — a well-reasoned exception. This is textbook-quality.
12. Idempotent operations — HIGH
Verdict: Not directly addressed. ral doesn't have a declarative ensure/apply primitive. This is arguably an ecosystem/convention concern rather than a language concern — idempotent commands are a property of the commands you call, not the shell. But a --dry-run or -WhatIf equivalent at the shell level would help. The _audit tree gives agents enough info to reason about idempotency, and with enables mocking for preview, but there's no first-class "preview this operation" mode.
13. Consistent, portable behavior — MEDIUM
Verdict: Met by design. ral is explicitly not POSIX (RATIONALE). There's one shell with one behavior. No bashisms-vs-dash problem. The module system (use) and single configuration file (~/.ralrc) eliminate the .bashrc/.bash_profile matrix. Cross-platform behavior depends on the implementation, but the spec doesn't have platform-conditional semantics.
14. Module / namespace system — MEDIUM
Verdict: Fully met. use returns a map of bindings, _-prefixed names are excluded, results are cached, circular references are detected (§8). This is clean and sufficient. No package manager, but that's a tooling concern.
15. Comprehensive audit trail — MEDIUM
Verdict: Substantially met. _audit captures the full execution tree with source positions, arguments, status, stdout, stderr, return values, and children (§10.3). ral --audit provides script-wide auditing. MCP serialization is specified. The tree records intermediate return values. What's missing: identity/principal information (who ran it), timestamps, and decision context (why the agent chose this command). These are better handled by the agent framework, as the RATIONALE implicitly acknowledges.
16. Good debugging experience — MEDIUM
Verdict: Partially met. The audit tree is a powerful post-mortem tool. Error messages report mismatch, expected type, received value, source location, and hints (§10.5). But there's no step-through debugger in v1 (§10.4 explicitly says so). No breakpoints, no variable watches. The _audit tree is better than set -x but not a replacement for a real debugger.
17. Rich interactive UX — MEDIUM
Verdict: Specified but implementation-dependent. Line editing, history, completion, and prompt rendering are listed (§18) but delegated to the host language. Aliases are interactive-only. ~/.ralrc supports prompt customization and edit mode. There's no mention of syntax highlighting or autosuggestions — Fish's signature features. This is adequate for v1 but not a differentiator.
18. --dry-run for destructive operations — MEDIUM
Verdict: Not met. No -WhatIf equivalent. with can mock commands, which is a building block for preview, but there's no built-in "show what this would do" mode. An agent could use _audit with mocked commands, but that's a manual composition, not a first-class feature.
19. Taint tracking for untrusted input — MEDIUM
Verdict: Not met. No taint mode. from-X decoders and ask don't mark values as tainted. External input is indistinguishable from program-constructed data. The static effect analyzer (ral-check, v1.1) could potentially track this, but it's not in the current spec.
20. Natural language integration — MEDIUM
Verdict: Not addressed. No NL-to-command translation. The MCP endpoint enables agent use, but there's no built-in "explain this command" or "translate this intent" feature. Reasonable to omit — this is better handled by the agent layer.
21. Parallel execution primitives — MEDIUM
Verdict: Substantially met. spawn, await, race, cancel, par with concurrency limit, disown (§13). Isolated child model with no shared mutable state. par is basically Nushell's par-each. No automatic parallelization (PaSh/POSH style), but the explicit primitives are solid.
22. Sane configuration model — MEDIUM
Verdict: Fully met. Single file: ~/.ralrc. Returns a configuration map with typed keys. Aliases are interactive-only and don't affect scripts. No .bash_profile vs .bashrc confusion.
23. First-class Unicode support — LOW
Verdict: Substantially met. ral is UTF-8 only. Invalid sequences in byte input become U+FFFD. String operations work on UTF-8. Paths are strings (no byte-path ambiguity). The RATIONALE notes binary data stays in byte pipes and doesn't enter the value space.
24. Performance / no fork-per-command — LOW
Verdict: Partially met. External commands still fork (unavoidable). Value pipelines don't use OS pipes — they're in-process handoff. Tail-call optimization is guaranteed. Builtins run in-process. But the spec doesn't mention threading for builtins (as Murex does) or particularly fast startup. Reasonable for v1.
25. Decision traces for AI explainability — LOW
Verdict: Partially met. The _audit tree records what happened but not why. No decision context, no alternative-path recording. The MCP endpoint provides the tree automatically, which is better than most shells offer. The gap is reasonable — decision traces belong in the agent framework.

Summary scorecard: Of 25 desiderata, ral fully meets 9, substantially meets 8, partially meets 4, and doesn't meet 4. The unmet items cluster around runtime security enforcement (#5, #10, #19) and operation preview/rollback (#9, #12, #18). The fully-met items are exactly the foundational ones: word splitting, error handling, static parsing, scoping, quoting, configuration, and modules.

Suggestions for improvement, in priority order:
1. Bring sandbox forward from "future" to v1.1 alongside ral-check. The capability gap is the single biggest weakness for the agentic use case. Even a minimal version — deny-all default, allowlist of commands and paths, network egress control — would put ral ahead of every existing shell. The with mechanism already provides the lookup-override machinery; sandbox would add enforcement teeth. Without it, the spec's agentic safety claims rest entirely on the agent framework, not the shell.
2. Add a --dry-run / --plan mode. When ral --plan script.ral runs, every external command is replaced with a no-op that logs what would have been executed (command, arguments, redirects, environment) as a structured tree. This gives agents a preview mechanism without requiring manual with mocking for every command. PowerShell's -WhatIf is the model. Implementation cost is low given that with and _audit already exist.
3. Consider taint tracking in ral-check. The v1.1 static effect analyzer is the natural place to add data-flow tracking for untrusted input. Values originating from ask, from-X decoders, $env, and $args could be marked as tainted. The analyzer could warn when tainted values reach _fail, command arguments (especially in head position via interpolation), or to-X encoders without passing through a validation function. This would be the first shell to replicate Perl's taint mode statically.
4. Add timestamps to _audit tree nodes. The execution tree already has cmd, args, status, stdout, stderr, value, and source location. Adding start_time and end_time (epoch milliseconds) per node would make the tree useful for performance profiling and compliance auditing with zero additional API surface.
5. Specify a --check or --validate flag for external commands. A convention (not enforcement) where ral --plan looks for a --dry-run flag on external commands. If the command supports it, use it; if not, log the command without executing. This bridges the gap between shell-level preview and command-level idempotency.
6. Consider streaming value pipelines for v2. The current design makes value stages eager, which is correct and simple. But for large datasets (processing millions of JSON records), eager materialization is a bottleneck. A future version could add lazy/streaming value stages, perhaps via an explicit stream adapter, without changing the default eager semantics.
7. The first_ok combinator from RATIONALE should be in the prelude. It's mentioned as the pattern for "try each item until one succeeds" but doesn't appear in §17. Same for collect_ok and for_ok. These are high-value for agent scripts that try multiple fallback strategies.
